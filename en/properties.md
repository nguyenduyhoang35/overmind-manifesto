# Overmind — Properties

> These are not rules. These are natural properties that emerge when following the Overmind manifesto.
> You don't need to strive for them — they appear when the principles are applied correctly.
> If you don't see them emerging, that's a signal that something in how you're applying needs review.

---

## Self-Improving

The system improves itself over time without changing the model or tools. Each execution produces feedback, feedback becomes knowledge, knowledge improves the next execution.

Quality is a function of knowledge completeness, not model capability. Same AI, same model — iteration 0 yields 84% quality, iteration 21 yields 100% — the difference comes entirely from accumulated knowledge.

## Failure-Positive

Mistakes have positive value because each failure generates new knowledge. The system never fails the same way twice. Failing early is cheaper than failing late. A rough first iteration is a good signal — the system is learning fast.

Corollary: the team doesn't need to fear mistakes. They need to fear mistakes that don't capture lessons.

## Compounding Knowledge

Knowledge accumulates and compounds over time. Constraints from iteration 0 still hold value at iteration 21. Knowledge doesn't depreciate like code or tools — it only accumulates or gets refined.

The more loops, the more valuable the knowledge store becomes. Returns accelerate over time, not decelerate.

## Transferable Wisdom

A new team member reads the knowledge store = receives accumulated experience from every prior iteration without going through them. Knowledge separates from individuals — it's not lost when someone leaves the team.

Onboarding becomes "read the knowledge store" instead of "sit next to a senior for 3 months."

## Emergent Quality

Quality doesn't come from process enforcement but emerges from knowledge richness. When knowledge is comprehensive enough, output is naturally correct without needing multiple review layers.

Complex processes are a symptom of missing knowledge. When you find yourself adding more gates and checkpoints, ask: "what knowledge is missing that's causing the output to be wrong?"

## Knowledge > Contributor

The knowledge store is smarter than any individual contributor. Humans provide domain truth, AI cross-references with existing knowledge, the result is more refined than the original input.

The store accumulates everyone's perspectives over time — BAs contribute business rules, developers contribute technical constraints, testers contribute failure patterns. The whole is greater than the sum of its parts.

## Tool-Agnostic

Value lies in the knowledge, not the tool. Switch execution engines — the knowledge store retains its value. Tools are replaceable, knowledge is durable.

Corollary: don't invest too heavily in tool-specific optimization. Invest in knowledge quality — it's portable wherever you go.

## Domain-Agnostic

The same closed loop works for software, marketing, operations, research, any domain. The only difference: which knowledge types matter and how to discover them.

Software needs business rules and technical constraints. Marketing needs audience insights and brand voice. Operations needs process rules and exception handling. The closed loop is the same — knowledge → execute → feedback → knowledge.

## Organic Complexity

Complexity only appears when pain demands it, not from upfront design. Start with 1 folder, 3 files. Add skills when you see repeating patterns. Add tooling when scale demands it. Each step of complexity has justification from real pain.

Corollary: two teams applying Overmind will have different tooling because their pain points differ. That's correct, not inconsistency.

## Diminishing Error Rate

Error rate decreases asymptotically with knowledge accumulation. Large improvements come early — the first few constraints eliminate many common errors. It gradually plateaus as knowledge covers most cases.

You never reach 0% error — but each new error becomes the last of its kind.

## Portable Across Projects

Applicable knowledge carries over to new projects. Constraints, conventions, decisions, failure patterns — many things transcend a specific project.

Project #2 doesn't start from zero. It starts from a subset of proven knowledge. A head start instead of a cold start.

## Scale-Invariant Principle

The same closed loop works for 1 person or 100. Larger team = more knowledge contributors = richer store = better output.

No need to add management layers when scaling. Need to add governance for the knowledge store — but the principle doesn't change.
